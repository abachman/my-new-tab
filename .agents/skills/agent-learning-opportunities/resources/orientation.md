# Repo Orientation: my-new-tab

> Generated by /orient. Re-run to update.

## One-line purpose
A custom Chrome/Firefox new tab page built with React and Redux, displaying a user-configurable grid of "blocks" (clocks, weather, links, templates) that persist to browser extension storage.

## Primary language(s)
JavaScript (React/JSX), with CSS stylesheets

## Pipeline / workflow stages
1. **Hydration** — on load, `store.js:initStore` reads persisted state from browser storage via `lib/storage/index.js:hydrate`, merging with `DefaultState` to fill in any missing keys
2. **Store initialization** — a Redux store is created with the hydrated state and `INITIALIZE_EVERYTHING` is dispatched
3. **Render** — `src/index.jsx` wraps `<App>` in a Redux `<Provider>` and renders into `#root`
4. **Block display** — `Views/BlocksGrid.jsx` maps `state.blocks` to typed block components (Clock, Weather, Link, Template, Bare) inside a responsive drag-and-drop grid
5. **User interaction** — `Views/Controls.jsx` dispatches actions to toggle editing/compacting mode and open forms; `Views/Forms/` renders modals for creating/editing blocks and settings
6. **Persistence** — every store state change triggers `lib/storage/index.js:persist`, writing the full state back to browser storage under the key `__my_new_tab`
7. **Export/Import** — `lib/storage/exportImport.js` handles JSON backup downloads and validated file imports via `IMPORT_DATA` action, which replaces the entire Redux state

## Core concepts

**Blocks**: The fundamental unit of the UI. Each block is a plain object stored in `state.blocks` keyed by a generated ID. It has a `type` field (`clock`, `weather`, `link`, `template`, `userscript`) and type-specific config fields. `BlocksGrid.selectView()` switches on `block.type` to choose the rendering component.

**Redux state shape**: Defined by `DefaultState` in `src/reducers/index.js`. Key slices: `blocks` (user content), `layouts` (grid positions per breakpoint), `layout` (editing/compacting mode flags), `weathers` (cached weather data), `forms` (modal visibility). This is the single source of truth — all UI state lives here.

**Hydration/Persistence cycle**: The app never reads from storage reactively. Instead, on startup `hydrate()` reads and merges storage into the initial Redux state. After that, the store subscriber calls `persist()` on every state change. This means browser storage is always slightly behind the in-memory store.

**Responsive grid layout**: `react-grid-layout` manages block positions. Columns and breakpoints are computed dynamically in `BlocksGrid` using a `STEP=80` pixel increment. Grid state (positions per breakpoint) is stored in `state.layouts` and also persisted per-block in `block.coords`.

**Block forms**: Creating or editing a block opens a modal form via `TOGGLE_BLOCK_EDITOR`. Form components live in `src/Views/Forms/Blocks/` with one file per block type. The forms are orchestrated by `src/Views/Forms/Blocks/index.jsx` which selects the right sub-form based on `forms.blocks.form.type`.

## Common gotchas

- **`src/index.jsx` imports from `"store"` not `"./store"`** — this works because `vite.config.js` has alias resolution. If you add new root-level lib imports (like `"actions"`, `"stylesheets/..."`) you need to check the vite aliases, not just the file path.

- **`IMPORT_DATA` bypasses all individual reducers** — it's handled in `rootReducer` before `combinedReducer` runs, replacing the entire state with `{ ...DefaultState, ...action.payload }`. Individual reducer logic does not run on import. If you add a new state slice, you must ensure its default value is in `DefaultState` or imports from older backups will leave it undefined.

- **Weather data is cached in Redux state, not re-fetched on every render** — `Actions.getWeather` checks `state.weathers[block.id].loadedAt` and skips the network call if it's less than 5 minutes old. If weather appears stale during development, this cache is the reason.
